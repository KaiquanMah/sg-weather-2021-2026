### 7. PyArrow Timestamp Conversion Errors (Missing Timezone)
**The Issue:**
* BigQuery's strict `pyarrow` ingestion engine crashed because the raw ISO format string (`2026-03-07T09:35:51.811832`) was missing an explicit UTC/Zulu offset, leading to unrecognized pandas objects.
* BigQuery is complaining about the timestamp strings missing a timezone identifier (e.g. 2026-03-07T09:35:51.811832).
* When Python generates datetime.utcnow().isoformat(), it doesn't always automatically append the Z (Zulu/UTC) timezone indicator by default unless the datetime object is explicitly made timezone-aware. BigQuery requires this trailing Z or a strict offset (like +00:00) when reading strings mapped to a Timestamp schema column.
**The Fix:**
* I rewrote the timestamp generation to use `datetime.now(pytz.timezone('Asia/Singapore')).isoformat()` to inject the explicit `+08:00` offset, and forcefully casted the payload column using `pd.to_datetime` before BigQuery ingestion.
* I have updated both api_client.py and loaders.py to correctly format all ingest_timestamp strings to end with Z (e.g., 2026-03-07T09:35:51.811832Z).
* I have just modified scripts/api_client.py to use Python's pytz timezone library:
```python
"ingest_timestamp": datetime.now(pytz.timezone('Asia/Singapore')).isoformat()
```
* This will automatically inject the `+08:00` UTC offset into the ISO string for BigQuery to parse natively as Singapore Time, solving the "missing zone offset" error
* Because you recently added pandas-gbq and pyarrow to the dependencies (which is perfect, as it is incredibly fast!), the loading engine is now strictly enforcing data types before sending the payload to BigQuery.
* Previously, the timestamps were stored in the pandas dataframe natively as string objects ("2026-03-07T09:35:51.811832"). pyarrow is complaining because it expects actual datetime objects.
* I've fixed this in scripts/loaders.py by converting those string columns explicitly into timezone-aware pandas datetime columns before BigQuery loading:
```python
if 'timestamp' in df.columns:
    df['timestamp'] = pd.to_datetime(df['timestamp'])
if 'ingest_timestamp' in df.columns:
    df['ingest_timestamp'] = pd.to_datetime(df['ingest_timestamp'])
```

### 9. Why Rate Limits and Timeout Errors Decreased After Parallelization
**The Issue:**
* You noticed that after parallelizing, the pipeline stopped experiencing timeout errors. This seemed counter-intuitive.


**The Explanation 1:**
* It seems counter-intuitive that sending more requests in parallel would result in fewer rate limits, but it comes down to how requests handles timeouts vs how your request_delay is structured!
* The Old Sequential Way
  * Previously, the code was looping through the 4 endpoints one by one. For each endpoint, it would:
    * Make an API request
    * Wait for the server to process and return the JSON payload
    * Call time.sleep(1.5) (your Config.REQUEST_DELAY)
    * Move on to the next endpoint
  * Because it was sequential, the script spent a lot of time doing absolutely nothing while waiting for the server to slowly build and return each JSON response. If the Data.gov.sg servers were slightly slow that day, the connection would hang open for a long time, eventually triggering a requests Timeout Error.
* The New Parallel Way
  * When we wrapped the logic in a ThreadPoolExecutor(max_workers=4), the script now does this:
    * It spins up 4 separate worker threads simultaneously.
    * Thread 1 asks for air-temperature, Thread 2 asks for relative-humidity, etc... at the exact same millisecond.
    * Crucially, the 1.5 second sleep timer now happens inside each thread independently.
    * Because the 4 requests are fired practically simultaneously, they count as a very brief "burst" against the API's limit of 30 requests/10s. The burst is only 4 requests—well under the limit!
  * By the time the script moves to the next date (e.g., from 2020-12-01 to 2020-12-02), it has already spent time processing the data payload, uploading to GCS, and uploading to BigQuery. This natural "computational delay" between days, combined with the 1.5 second sleep inside the threads, perfectly spaces out the burst of 4 requests, making sure you never hit the 30 req/10s ceiling.
* The True Culprit: Timeouts vs Rate Limits
  * What you were experiencing previously wasn't actually an explicit HTTP 429 Rate Limit Exceeded (which my retry logic catches and waits for). You were getting Timeout Errors where the connection just died.
  * By parallelizing the requests, you drastically reduced the total duration the script keeps connections open. Instead of keeping 1 connection open for a long time sequentially, you open 4 connections, grab the data fast, and close them immediately. This makes the ingestion significantly more robust against network hiccups!


**The Explanation: 2**
* Previously, the code hit sequential endpoints:
    1. HTTP request for endpoint A
    2. Wait a long time for the server...
    3. Sleep 1.5 seconds.
    4. HTTP request for endpoint B.

Because it was sequential, the script spent excessive time waiting, leaving single open connections dangling. If the servers were slow, `requests` would kill the connection with a Timeout.

With Parallelization (`ThreadPoolExecutor`):
1. It spins up 4 distinct worker threads instantly.
2. All 4 request their respective payloads simultaneously.
3. The 1.5s sleep occurs entirely within each independent thread.

Because the limit is 30 requests/10s across APIs, our burst of 4 instantaneous requests easily stays under the limit. Furthermore, the natural CPU delay between each loaded date (and BigQuery processing), combined with the internal 1.5s thread sleep, spaces out these independent bursts perfectly—meaning connections open quickly, fetch fast, and close cleanly, eradicating the prior Sequential Timeout hanging behavior.
